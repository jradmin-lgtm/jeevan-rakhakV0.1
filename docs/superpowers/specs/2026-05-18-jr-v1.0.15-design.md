## 2. Server changes

### 2.1 Migrations (additive, run in `bootstrap()`, fail-loud per ops thumb-rule)

```sql
CREATE TABLE IF NOT EXISTS driver_locations (
  driver_id   uuid PRIMARY KEY REFERENCES drivers(id) ON DELETE CASCADE,
  lat         double precision NOT NULL,
  lng         double precision NOT NULL,
  updated_at  timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_driver_locations_updated_at ON driver_locations(updated_at);

CREATE TABLE IF NOT EXISTS sos_dispatch_attempts (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  booking_id  uuid NOT NULL REFERENCES bookings(id) ON DELETE CASCADE,
  driver_id   uuid NOT NULL REFERENCES drivers(id) ON DELETE CASCADE,
  wave_number int NOT NULL,
  distance_km numeric(6,2),
  pushed_at   timestamptz NOT NULL DEFAULT now(),
  rejected_at timestamptz,
  accepted_at timestamptz,
  UNIQUE (booking_id, driver_id)
);
CREATE INDEX IF NOT EXISTS idx_sos_attempts_booking ON sos_dispatch_attempts(booking_id);

ALTER TABLE bookings ADD COLUMN IF NOT EXISTS paid_inr    int;
ALTER TABLE bookings ADD COLUMN IF NOT EXISTS paid_at     timestamptz;
ALTER TABLE bookings ADD COLUMN IF NOT EXISTS paid_coupon text;

-- Existing completed pilot bookings count as paid-₹0
UPDATE bookings SET paid_at = COALESCE(updated_at, created_at), paid_inr = 0
 WHERE status='COMPLETED' AND paid_at IS NULL;
```

### 2.2 New endpoints

- `POST /api/v1/driver/heartbeat { lat, lng }` — auth JWT. Upserts `driver_locations`. Returns 204.
- `POST /api/v1/bookings/:id/mark-paid { couponCode? }` — auth JWT, must match `booking.userId`.
  - Validates `status==='COMPLETED'` and `paid_at IS NULL`.
  - Computes discount (PILOT100 = 100%-off for now), stores `paid_inr`/`paid_at`/`paid_coupon`.
  - Emits `booking:paid` over socket.
  - Returns full breakdown + `paid` block. Idempotent on repeat (second call returns existing payment).
- `POST /api/v1/bookings/:id/reject` (driver) — for SOS cascade. Inserts/updates `sos_dispatch_attempts.rejected_at`. Cascade engine removes driver from future waves.

### 2.3 Cascade engine (api-server in-memory + Postgres audit)

`services/api-server/src/sos-cascade.ts` exports `startCascade(bookingId)` and `stopCascade(bookingId)`.

Algorithm:
1. On `POST /bookings` with `emergencyType=SOS`, call `startCascade(bookingId)`.
2. Eligible list: `SELECT FROM driver_locations WHERE updated_at > now() - INTERVAL '5 min'` joined to online active drivers, distance-sorted (Haversine, JS-side), top 10.
3. Wave 1 (t=0): insert attempt row for driver #1, emit `sos:incoming` via socket-server internal API, schedule wave 2 at t+60s.
4. Wave N (N∈2..10): for each driver i in 1..N not rejected/accepted: insert+emit if new, otherwise no-op. Schedule N+1.
5. On accept (any driver): atomic `UPDATE bookings SET status='ACCEPTED' WHERE id=? AND status='REQUESTED' RETURNING ...`. Winner gets routed to TripScreen, losers get `sos:cancelled`. Cascade stops.
6. On reject: mark attempt row, driver dropped from future waves, cascade continues.
7. On cap reached (post-wave-10, t≈600s): fire critical system_event, emit `sos:cascade_exhausted` to patient, booking stays REQUESTED for manual ops handling.
8. Empty eligible list at t=0: same cascade_exhausted state immediately.

Restart resilience: on api-server boot, scan `bookings WHERE status='REQUESTED' AND emergencyType='SOS' AND created_at > now() - 10 min`, rebuild `CascadeRunner` for each, resume from next wave.

Configurable: `SOS_CASCADE_MAX_DRIVERS=10`, `SOS_CASCADE_WAVE_INTERVAL_S=60`, `SOS_CASCADE_STALENESS_MIN=5`.

### 2.4 Booking creation flow (SOS branch)

Currently: `POST /bookings` returns booking + driver app sees it in `/bookings/pending`. Driver dashboard polls + sorts by distance.

After v1.0.15:
- `emergencyType=SOS`: skip the public pending pool. Server starts cascade. Driver dashboard does NOT show SOS bookings in its general list. Drivers only see SOS via the pushed `sos:incoming` modal.
- Other types: existing behavior unchanged.

## 3. Mobile changes

### 3.1 New screens

**`apps/user-app/src/screens/PaymentScreen.tsx`** — appears post-COMPLETED for SOS only.
- Triggered by `LiveTrackingScreen` navigating on `status==='COMPLETED' && !paid_at && emergencyType==='SOS'`.
- Renders shared `<FareBreakdown>` (extracted from `BookAmbulanceScreen` into `packages/ui`).
- Coupon input pre-filled `PILOT100`, "Apply" → discount row + `YOU PAY ₹0`.
- "Mark paid · finish" → `POST /mark-paid` → `navigate.replace('Rating', { bookingId })`.
- Android hardware back disabled (force user through the flow). Re-entry safe on app relaunch.

**`apps/driver-app/src/screens/TripHistoryScreen.tsx`** — replaces `EarningsScreen.tsx`.
- Lists completed bookings (`bookings.mine()` filtered to `COMPLETED`).
- Columns: date · pickup→drop addresses (truncated 24ch) · km (Haversine pickup→drop) · time taken (`completedAt − acceptedAt`).
- No money.
- Dashboard's `onEarnings` callback rewired to `onTripHistory`.

**`apps/driver-app/src/components/SosIncomingModal.tsx`** — fullscreen modal over Dashboard.
- Subscribed to socket `sos:incoming` and `sos:cancelled`.
- Renders: red pulse animation, "EMERGENCY · {type}", distance + ETA, ACCEPT + REJECT buttons.
- Accept → `POST /bookings/:id/accept` → navigate to TripScreen on 200, dismiss on 409.
- Reject → `POST /bookings/:id/reject` → dismiss modal.

### 3.5 Heartbeat client

`apps/driver-app/src/hooks/useDriverHeartbeat.ts`:
- Active when `driver.online === true && AppState === 'active'`.
- Every 60s: get GPS via `expo-location`, `POST /driver/heartbeat`.
- Pauses on background, resumes on foreground.
- Stops cleanly when driver toggles offline.

## 5. Risk register

| Risk | Mitigation |
|---|---|
| Driver heartbeat battery drain | Pause when backgrounded; only ping when online toggle is on. |
| Cascade timer dies on api-server restart | Boot-time scan + resume. Worst case: single 60s gap in cascade. |
| Hindi malfunction broader than Picker hypothesis | Time-box investigation to 2h; ship partial Hindi if needed. |
| 10-driver cap insufficient | `SOS_CASCADE_MAX_DRIVERS` env var, default 10. |
| `mark-paid` races with `complete-trip` | Status check returns 409 if not COMPLETED yet; client retries on socket event. |
| Render auto-deploy flake (per CONTEXT.md ops rule 2) | Verify Render commit after push + force-redeploy via API. |
